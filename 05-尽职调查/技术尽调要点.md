# 技术尽调要点

## 概述
技术尽职调查（Technical Due Diligence, Tech DD）是对目标公司的技术架构、代码质量、技术团队能力和技术资产进行系统性评估的过程。对于科技类投资标的，技术尽调直接关系到估值合理性、技术风险识别和投后赋能方向。它回答的核心问题是：这家公司的技术是否真正构成了竞争壁垒，以及技术资产是否可持续。

## 核心内容

### 一、技术架构评估

#### 架构审查框架

| 评估维度 | 评估要点 | 评分标准 |
|---------|---------|---------|
| 整体架构设计 | 微服务/单体/混合架构的合理性 | 是否匹配业务规模和团队能力 |
| 可扩展性 | 水平扩展、垂直扩展能力 | 是否支持10倍增长无需重构 |
| 可用性 | SLA目标、容灾设计 | SLA是否达到99.9%以上 |
| 性能 | 响应时间、吞吐量、并发能力 | 是否满足业务峰值需求 |
| 安全性 | 认证授权、数据加密、安全审计 | 是否通过等保/ISO27001 |
| 可维护性 | 模块化程度、文档完整性 | 新人上手时间、代码可读性 |

#### 架构评估 Checklist

- [ ] 获取系统架构图（包括前端、后端、数据层、基础设施层）
- [ ] 了解技术栈选型及选型理由
- [ ] 评估微服务拆分的粒度是否合理
- [ ] 检查API设计规范和版本管理策略
- [ ] 审查数据库选型和数据模型设计
- [ ] 评估缓存策略和消息队列使用情况
- [ ] 检查CI/CD流水线的成熟度
- [ ] 审查监控告警体系的覆盖度
- [ ] 评估容器化和编排方案（Docker/K8s等）
- [ ] 检查多云/混合云策略（如适用）

#### 常见架构模式评估

| 架构模式 | 适用场景 | 优势 | 风险 |
|---------|---------|------|------|
| 单体架构 | 早期产品、小团队 | 开发快、部署简单 | 扩展困难、技术债累积 |
| 微服务 | 大规模系统、多团队 | 独立部署、技术多样性 | 运维复杂、分布式事务 |
| Serverless | 事件驱动、流量波动大 | 成本低、自动扩缩 | 冷启动延迟、供应商锁定 |
| 事件驱动架构 | 实时数据处理、解耦 | 松耦合、高扩展性 | 调试困难、最终一致性 |

### 二、代码质量评估

#### 代码质量指标

| 指标 | 定义 | 健康标准 | 工具 |
|------|------|---------|------|
| 测试覆盖率 | 测试代码覆盖的代码比例 | > 70%（核心模块 > 85%） | JaCoCo, Istanbul, Coverage.py |
| 代码重复率 | 重复代码占总代码比例 | < 5% | SonarQube, PMD |
| 圈复杂度 | 代码分支路径数量 | 单函数 < 15 | SonarQube, CodeClimate |
| 技术债务比率 | 修复所有问题所需时间/开发总时间 | < 5% | SonarQube |
| 代码规范合规率 | 符合编码规范的代码比例 | > 95% | ESLint, Checkstyle |
| 依赖漏洞数 | 存在已知安全漏洞的依赖包数 | 0个高危/严重 | Snyk, Dependabot |

#### 代码审查 Checklist

- [ ] 抽样审查核心模块的代码质量（至少覆盖3-5个核心模块）
- [ ] 检查代码规范（命名、注释、格式）的执行情况
- [ ] 评估单元测试、集成测试、端到端测试的覆盖情况
- [ ] 审查代码评审（Code Review）流程和执行记录
- [ ] 检查分支管理策略（Git Flow / Trunk-based）
- [ ] 评估静态代码分析工具的使用情况
- [ ] 核查第三方依赖的版本管理和安全更新策略
- [ ] 审查代码仓库的提交频率和贡献者分布

#### 开源合规审查

- [ ] 列出所有使用的开源组件及其许可证类型
- [ ] 检查是否存在Copyleft许可证（GPL、AGPL）的合规风险
- [ ] 验证开源组件的使用是否符合许可证要求
- [ ] 评估开源组件的安全漏洞情况
- [ ] 检查是否建立了开源组件管理流程（SCA工具使用）

### 三、技术团队能力评估

#### 团队评估框架

| 维度 | 评估要点 | 信息来源 |
|------|---------|---------|
| 团队规模 | 研发人员数量、占比 | HR数据 |
| 团队结构 | 技术栈分布、层级结构 | 组织架构图 |
| 核心人员 | CTO/Tech Lead的背景和能力 | 面谈、履历 |
| 技术文化 | 工程实践、学习氛围 | 团队访谈、代码仓库 |
| 人才留存 | 核心人员在职时间、离职率 | HR数据、面谈 |
| 招聘能力 | 招聘周期、offer接受率 | HR数据 |

#### 团队评估 Checklist

- [ ] 与CTO/技术VP进行深度技术对话（至少2小时）
- [ ] 与3-5名一线工程师进行匿名访谈
- [ ] 审查技术团队的组织架构和职责分工
- [ ] 评估核心技术人员的行业经验和专业深度
- [ ] 检查技术团队的稳定性（最近12个月离职率）
- [ ] 审查技术决策流程和技术债管理策略
- [ ] 评估团队的技术学习和创新能力
- [ ] 检查技术人才梯队建设情况

#### 关键人员风险评估

| 风险类型 | 识别信号 | 缓解措施 |
|---------|---------|---------|
| 关键人依赖 | 核心系统只有1-2人理解 | 知识文档化、交叉培训 |
| 竞业风险 | 核心人员未签署竞业限制 | 补签协议、股权绑定 |
| 团队分裂 | 技术方向分歧、派系对立 | 深入了解原因、评估影响 |
| 技能老化 | 技术栈陈旧、缺乏学习动力 | 评估技术升级计划 |

### 四、专利布局分析

#### 专利评估 Checklist

- [ ] 获取完整的专利清单（发明专利、实用新型、外观设计、PCT国际申请）
- [ ] 评估核心专利的技术覆盖范围和保护强度
- [ ] 分析专利的权利要求是否覆盖核心商业方法
- [ ] 检查专利的法律状态（有效、审查中、被驳回、被无效）
- [ ] 评估专利组合的防御价值（是否能阻止竞争对手）
- [ ] 分析专利布局的地理覆盖范围
- [ ] 检查是否存在专利诉讼风险（FTO分析）
- [ ] 评估专利维护成本与价值的匹配度

#### 专利价值评估矩阵

| 维度 | 高价值 | 低价值 |
|------|--------|--------|
| 技术相关性 | 直接覆盖核心产品/技术 | 与主营业务关联度低 |
| 权利要求范围 | 覆盖范围广、难以规避 | 范围窄、容易设计规避 |
| 有效期 | 剩余有效期长（>5年） | 即将到期或已被部分无效 |
| 地域覆盖 | 多国申请（中/美/欧/日） | 仅在一国申请 |
| 实施情况 | 已在产品中实施 | 仅作为储备、未实施 |

### 五、技术债务评估

#### 技术债务分类

| 类型 | 描述 | 影响程度 | 偿还优先级 |
|------|------|---------|-----------|
| 设计债务 | 架构设计不合理、模块耦合度高 | 高 | 高 |
| 代码债务 | 代码质量差、缺乏测试、文档缺失 | 中-高 | 中 |
| 依赖债务 | 过时的框架/库、存在安全漏洞 | 中-高 | 高 |
| 基础设施债务 | 手动部署、缺乏监控、服务器老旧 | 中 | 中 |
| 文档债务 | 架构文档缺失、API文档不完整 | 中 | 低 |
| 测试债务 | 测试覆盖不足、缺乏自动化测试 | 高 | 高 |

#### 技术债务评估 Checklist

- [ ] 使用SonarQube等工具进行代码质量扫描
- [ ] 统计TODO/FIXME/HACK注释的数量和分布
- [ ] 评估核心系统的平均故障间隔时间（MTBF）和恢复时间（MTTR）
- [ ] 检查是否有技术债务偿还计划（Tech Debt Backlog）
- [ ] 估算偿还核心技术债务所需的时间和资源
- [ ] 评估技术债务对产品迭代速度的影响
- [ ] 检查历史技术决策的文档记录

#### 技术债务量化估算

```
技术债务成本 = 修复时间 × 工程师日薪 + 机会成本（延误的新功能价值）

示例估算：
- 高优先级债务修复：3人月 × 3万/月 = 9万元
- 中优先级债务修复：6人月 × 3万/月 = 18万元
- 低优先级债务修复：12人月 × 3万/月 = 36万元
- 总技术债务：约63万元（不含机会成本）
```

### 六、数据安全与隐私合规

#### 数据安全评估 Checklist

- [ ] 审查数据分类分级制度及执行情况
- [ ] 检查数据加密措施（传输加密、存储加密）
- [ ] 评估访问控制机制（RBAC、最小权限原则）
- [ ] 审查数据备份和恢复策略
- [ ] 检查数据安全事件响应流程
- [ ] 评估第三方数据处理的合规性（数据处理协议DPA）
- [ ] 核查个人信息处理的合法性基础
- [ ] 检查数据跨境传输的合规性
- [ ] 审查用户隐私政策的完整性和准确性
- [ ] 评估数据泄露通知机制的完善程度

#### 隐私合规要求对照

| 法规 | 核心要求 | 合规要点 |
|------|---------|---------|
| 《个人信息保护法》 | 合法、正当、必要、诚信原则 | 知情同意、最小必要、目的限制 |
| 《数据安全法》 | 数据分类分级、安全审查 | 重要数据识别、安全评估 |
| 《网络安全法》 | 网络安全等级保护 | 等保测评、安全管理制度 |
| GDPR（如涉及欧盟） | 数据主体权利、DPO设置 | 数据处理记录、影响评估 |

### 七、DevOps与工程效率

#### 评估 Checklist

- [ ] 审查CI/CD流水线的成熟度和自动化程度
- [ ] 评估部署频率和部署成功率
- [ ] 检查基础设施即代码（IaC）的实践情况
- [ ] 审查监控和可观测性体系（日志、指标、追踪）
- [ ] 评估事故响应流程和事后复盘（Post-mortem）机制
- [ ] 检查开发环境、测试环境、生产环境的隔离情况
- [ ] 评估代码从提交到部署的平均时间（Lead Time）

#### DORA指标基准

| 指标 | 精英水平 | 高效水平 | 中等水平 | 低效水平 |
|------|---------|---------|---------|---------|
| 部署频率 | 按需（每天多次） | 每天-每周 | 每周-每月 | 每月-每半年 |
| 变更前置时间 | < 1小时 | 1天-1周 | 1周-1月 | 1月-6月 |
| 变更失败率 | 0-15% | 16-30% | 16-30% | 46-60% |
| 事故恢复时间 | < 1小时 | < 1天 | 1天-1周 | 1周-1月 |

| 术语 | 英文 | 定义 |
|------|------|------|
| 技术债务 | Technical Debt | 为快速实现功能而做出的次优技术决策，需要未来偿还 |
| 微服务架构 | Microservices Architecture | 将应用拆分为一组小型、独立部署的服务的架构模式 |
| CI/CD | Continuous Integration / Continuous Deployment | 持续集成/持续部署，自动化软件交付流程 |
| SLA | Service Level Agreement | 服务级别协议，定义服务可用性和性能承诺 |
| FTO分析 | Freedom to Operate Analysis | 自由实施分析，评估产品是否侵犯他人专利权 |
| DORA指标 | DevOps Research and Assessment | 衡量软件交付效能的四个核心指标 |
| SCA | Software Composition Analysis | 软件成分分析，识别开源组件及其许可证和安全风险 |

| MTBF | Mean Time Between Failures | 平均故障间隔时间 |
| MTTR | Mean Time To Recovery | 平均恢复时间 |

## 实务要点

1. **技术尽调需要真正的技术人员参与**：投资团队中的非技术人员可以通过检查清单完成基础审查，但核心技术判断（架构合理性、代码质量、技术债务严重程度）必须由有经验的技术专家完成。建议聘请独立的技术尽调顾问。

2. **关注技术与业务的匹配度**：最好的技术架构是匹配业务需求的架构，而非最先进的架构。过度工程化（Over-engineering）和工程不足（Under-engineering）都是风险。评估技术选择是否与当前和可预见未来的业务规模匹配。

3. **技术债务需要量化**：不要仅用"代码质量一般"来描述技术债务。应该估算偿还债务需要的人月数、对产品迭代速度的影响、以及可能引发的系统故障风险。量化技术债务有助于在估值谈判中获得合理折扣。

4. **核心人员访谈比代码审查更重要**：与CTO和核心工程师的深度对话可以揭示代码和文档中看不到的信息：技术决策的背景、团队的真实能力、技术发展的方向。匿名访谈一线工程师可以发现管理层不愿透露的问题。

5. **数据安全是底线要求**：在数据监管趋严的背景下，数据安全和隐私合规不是加分项，而是底线要求。严重的数据安全问题可能导致巨额罚款、业务中断和声誉损失，直接影响投资回报。

## 常见误区

1. **将技术栈先进性等同于技术实力**：使用最新的技术框架（如最新版React、Rust等）并不一定意味着技术实力强。过度追求新技术可能导致人才招聘困难、生态不成熟、迁移成本高等问题。成熟、稳定的技术栈在很多场景下是更好的选择。

2. **忽视技术债务的累积效应**：单个技术债务看起来影响不大，但累积起来可能导致系统脆弱、迭代速度下降、人才流失。技术债务的增长往往是非线性的——当超过某个临界点后，修复成本会急剧上升。

3. **过度关注专利数量而非质量**：专利数量多不代表技术壁垒强。很多公司为了补贴或政策优惠申请大量低质量专利（如实用新型、外观设计），这些专利几乎没有防御价值。应关注发明专利的质量、技术相关性和保护范围。
